# План фронтенда

Контекст и домен — в [booking-service.md](booking-service.md). Контракт — `contract/main.tsp`
(TypeSpec, источник правды, не менять; OpenAPI генерится в `contract/tsp-output/openapi.yaml`).
Этот документ — про реализацию фронта под контракт **v2.0.0** (модель «типов событий»).

> **Контракт переехал** с «календарной» модели (помесячный календарь, глобальная длина
> слота, статусы слотов) на модель **типов событий**: у каждого типа своя длительность,
> доступность считается на фикс-окно 14 дней и отдаётся одним запросом (только свободные
> слоты). Фронт переписывается с нуля по новому контракту; от старого остаётся каркас
> (`api/client.ts` с Bearer-перехватчиком, `useConfig`, структура pages/components/hooks,
> MSW-инфраструктура, роутинг `/` + `/admin`).

## Стек

- React + Vite + TypeScript, пакетный менеджер npm.
- API: типы из контракта через `openapi-typescript`, запросы через `openapi-fetch`
  (типобезопасный fetch-клиент). Скрипт `npm run gen:api` сперва компилирует TypeSpec
  (`gen:contract`), затем генерит типы из `contract/tsp-output/openapi.yaml`.
- Даты/таймзоны: `date-fns` + `date-fns-tz` (форматирование по IANA-имени владельца).
- Тесты: Vitest + React Testing Library + MSW.
- Сгенерённые типы API **не коммитим** (генерятся скриптом), добавить в `.gitignore`.

## Экраны

### Публичный (гость) — по публичной ссылке, один `VisitorPage` с локальной state-машиной

Шаги хранятся в `useState` (`selectedTypeId`, `selectedDate`, `bookingSlot`), без под-роутов.

1. **Список типов событий** (`GET /event-types`): название, описание, длительность.
   Сверху — глобальный баннер владельца из `useConfig` (`ownerTimeZone` + рабочие часы),
   показывается ещё до выбора типа. Выбор типа → шаг 2.
2. **Доступность типа** (`GET /event-types/{id}/availability`) — один запрос, в ответе
   всё окно «сегодня…сегодня+`horizonDays`» (14 дней): все дни окна, у каждого `date`
   (в TZ владельца), `hasFreeSlots` и уже вложенный массив **только свободных** `slots`
   (`startAt`/`endAt`). Помесячной догрузки и навигации нет.
3. **Календарные сетки** (компонент `AvailabilityCalendar`, заменяет старый `DayGrid`):
   полноценная месячная сетка с подсвеченным 14-дневным окном. Если окно пересекает
   границу месяцев — показываем **две** месячные сетки с тем же подсвеченным окном.
   Десктоп — сетки горизонтально, мобайл — вертикально. Скелет месяца (заполнение до
   полных недель) строим на клиенте; дни вне окна и дни с `hasFreeSlots:false` — погашены
   и неактивны; дни с `hasFreeSlots:true` — кликабельны и подсвечены; выбранный день
   выделен. **Навигации по месяцам нет** — это статичная подача фиксированного окна.
4. Клик по дню → `SlotList`: **только свободные** слоты выбранного дня (без статусов).
   Время каждого слота — **в двух TZ**: основное в `ownerTimeZone`, рядом мелко
   «у вас это HH:mm» (TZ браузера). Конвертация — `date-fns-tz`.
5. Клик по слоту → `BookingForm`: имя (req), ссылка на встречу (req, URL), описание (opt).
   Валидация на стороне фронта (+ обработка ошибок бэка). В запрос идёт
   **`eventTypeId` + `startAt`** (длительность берёт бэк из типа).
6. Успех (`201`) → короткий **toast** «Вы записаны» + возврат к доступности и **рефетч**
   доступности типа (забронированный слот исчезает). Кнопка submit дизейблится на время
   запроса — защита от дабл-сабмита на фронте.
7. Явно показываем, в каком часовом поясе живёт владелец (баннер + подписи у времени).

### Владелец (admin) — с токеном (хранить в `localStorage`, слать `Authorization: Bearer`)

После входа по токену — секции на одной странице: типы событий сверху, ниже календарь
встреч и список предстоящих встреч. На десктопе календарь и список встреч идут в две
колонки, на мобильных остаются вертикальным стеком.

- **Типы событий:** список (`GET /admin/event-types`) + форма создания
  (`POST /admin/event-types`): название, описание, длительность. Длительность —
  **`select` из кратных 15** (15, 30, … 120), чтобы невалидная кратность была невозможна
  в принципе (диапазон и кратность так гарантируются на клиенте; бэк всё равно проверяет).
- **Календарь встреч:** месячная сетка ближайшего окна с подсветкой дней, где есть брони;
  счётчик броней показывается справа внутри дня, чтобы не перекрывать число.
- **Предстоящие встречи:** единый плоский список всех типов (`GET /admin/bookings`),
  отсортированный по времени: имя гостя, ссылка, описание, `eventTypeName`, время
  (в двух TZ) + кнопка отмены (`DELETE /admin/bookings/{id}`).
- Где-то на экране — публичная ссылка на профиль (origin) для копирования.

## Время на фронте

- Конфиг (`GET /api/v1/config`) тянем один раз на старте через `useConfig`:
  `ownerTimeZone`, `workingHours`, `gridMinutes`, `horizonDays`. Используется для
  глобального баннера владельца (TZ + рабочие часы), доступного ещё на экране списка
  типов, до выбора конкретного типа.
- Слоты приходят как `OffsetDateTime` в смещении владельца. Показываем **в двух поясах**:
  основное время — в `ownerTimeZone` (согласовано с группировкой дней по `date`), рядом
  мелко — то же время в TZ браузера («у вас это HH:mm»). Так нет риска, что слот визуально
  «уедет» на соседний день относительно своего `date`.
- Сетку дней (окно 14 дней) строим по тому, что отдаёт бэк (`days[].date`); границы дней
  согласованы с ним (считаются в TZ владельца).

## Ошибки

Парсим `application/problem+json` (RFC 9457): по `errorCode` показываем понятные сообщения;
для валидации — мапим `errors[]` на поля формы.

- `409 SLOT_ALREADY_BOOKED` → toast «время только что заняли», перезапросить доступность типа.
- `404 EVENT_TYPE_NOT_FOUND` → тип удалён/не найден, вернуть к списку типов.
- `422` (`SLOT_IN_PAST` / `SLOT_OUT_OF_HORIZON` / `SLOT_NOT_ON_GRID`) → доступность устарела:
  перезапросить и показать сообщение.
- `401 UNAUTHORIZED` (админка) → сбросить токен из `localStorage` и вернуть к форме входа.
- `400 VALIDATION_FAILED` → разложить `errors[]` по полям формы; `MALFORMED_REQUEST` → общее
  сообщение.

## Тесты (Vitest + RTL + MSW)

- MSW мокает API строго по контракту v2.0.0 (фронт пишется без живого бэка).
- Покрыть: список типов и выбор; рендер доступности (одна и две месячные сетки) и подсветку
  14-дневного окна; дизейбл дней без свободных слотов; открытие дня и список только свободных
  слотов; форму брони и её валидацию (включая `eventTypeId` в запросе); обработку ошибок
  (409/422/404); admin — создание типа (валидация длительности через `select`), календарь
  встреч в двухколоночной десктопной раскладке со счётчиком справа, список встреч и отмену;
  вход/выход по токену и реакцию на `401`.

## Сборка и запуск

- `npm ci && npm run gen:api && npm run dev` — локальная разработка.
- `npm run test` — Vitest. `npm run build` — продакшен-сборка в `dist/`.
- `dist/` на этапе Docker-сборки копируется в статику Spring.

## Definition of Done (фронт-сессия)

- Оба экрана (посетитель и владелец) работают на MSW-моках по контракту.
- Vitest зелёный.
- `npm run build` проходит без ошибок.
