# e2e — сквозные тесты (Playwright)

Сквозные тесты сервиса «Запись на звонок» на TypeScript. Гоняются против реального
стека (Spring отдаёт UI и API + Postgres). Сценарии и стратегия —
в [`docs/plans/e2e-scenarios.md`](../docs/plans/e2e-scenarios.md).

## Два стенда

Тесты делают `TRUNCATE`, поэтому **владеют своей БД** и не должны бить по стенду,
в котором вы тыкаете руками. Стенды разведены и могут работать одновременно:

| Стенд            | compose            | app    | postgres        | назначение                |
|------------------|--------------------|--------|-----------------|---------------------------|
| дев (ручной)     | `compose.yaml`     | `:8081`| `:5432`, volume | потыкать руками, данные живут |
| e2e (тестовый)   | `compose.e2e.yaml` | `:8082`| `:5433`, tmpfs  | прогон тестов, чистый каждый раз |

e2e-модуль по умолчанию целится в e2e-стек (`:8082`/`:5433`).

## Принципы

- **Предусловия** — посев данных напрямую в Postgres (`src/db.ts`): контроль над `id`,
  временем и состоянием без зависимости от API.
- **Действия** — через интерфейс (Page Objects в `src/pages`), как у реального пользователя.
  API дёргаем напрямую только для вспомогательных задач (поиск свободного слота, эмуляция гонки).
- **Проверки** — сперва UI (тосты, списки, исчезновение слотов), затем БД как подтверждение
  состояния (факт записи/удаления брони, поля).
- **Изоляция** — перед каждым тестом `TRUNCATE` таблиц `event_types`/`bookings`
  (токен владельца в `app_settings` не трогаем). Поэтому тесты идут последовательно (`workers: 1`).
- **Время** — браузер принудительно в поясе владельца (`Asia/Yerevan`): тесты не зависят
  от пояса раннера, а слоты выбираются детерминированно (первый свободный слот окна).

## Запуск

```bash
cd e2e
npm ci
npx playwright install chromium   # один раз

# Вариант А — стек поднимается и гасится сам (как в CI):
E2E_MANAGE_STACK=1 npm test

# Вариант Б — стек держите сами (быстрее для повторных прогонов):
docker compose -f compose.e2e.yaml up -d --build --wait   # из корня репозитория
npm test
docker compose -f compose.e2e.yaml down -v                # когда закончили
```

Прочее: `npm run test:ui` — интерактивный режим, `npm run report` — HTML-отчёт,
`npm run typecheck` — проверка типов.

## CI

`globalSetup`/`globalTeardown` сами поднимают и гасят e2e-стек (`shouldManageStack()`
включается при `CI=1`). Достаточно установить зависимости и браузер и вызвать `npm test` —
дев-стек при этом не нужен.

## Конфигурация (env)

См. `.env.example`. Значения по умолчанию — изолированный e2e-стек:

| Переменная         | Назначение                                  | По умолчанию            |
|--------------------|---------------------------------------------|-------------------------|
| `E2E_BASE_URL`     | адрес приложения (UI + API)                 | `http://localhost:8082` |
| `PGHOST` …         | подключение к БД e2e-стека                  | `localhost:5433/booking`|
| `E2E_MANAGE_STACK` | управлять стеком из прогона (`1`/`0`/авто)  | авто (вкл. в CI)        |

## Структура

```
compose.e2e.yaml              # изолированный e2e-стенд (в корне репозитория)
e2e/
├─ playwright.config.ts        # workers:1, baseURL, timezoneId, global setup/teardown
├─ src/
│  ├─ env.ts                   # адреса и параметры БД из env (дефолты — e2e-стек)
│  ├─ db.ts                    # pg: truncate / seed* / read* / getOwnerToken
│  ├─ api.ts                   # config/availability + поиск свободного слота
│  ├─ time.ts                  # формат времени слота / номер дня
│  ├─ stack.ts                 # up/down e2e-стека + shouldManageStack()
│  ├─ fixtures.ts              # POM-фикстуры, токен владельца, авто-сброс БД
│  ├─ global-setup.ts          # (опц.) поднять стек + проверка готовности
│  ├─ global-teardown.ts       # (опц.) погасить стек
│  └─ pages/                   # Page Objects (visitor, admin)
└─ tests/                      # по одному файлу на сценарий 1–5
```
