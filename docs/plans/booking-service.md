# План: «Запись на звонок» (Booking Service)

Учебный проект — упрощённая вариация Cal.com для обучения разработке с AI-агентами.
Этот документ — единый источник правды по замыслу. Детали реализации — в
[backend.md](backend.md) и [frontend.md](frontend.md). Контракт API — в `contract/openapi.yaml`.

## 1. Суть

Владелец публикует доступное время для встреч (звонков), случайный посетитель по
публичной ссылке выбирает свободный 30-минутный слот и бронирует его. Авторизации,
личных кабинетов и интеграций с внешними календарями нет — функциональность намеренно
ограничена.

## 2. Акторы и сценарии

**Владелец** (один на инсталляцию, доступ по секретному токену):
1. Смотрит календарь (месячная сетка), видит, в какие дни есть встречи.
2. Открывает день, разбитый на 30-мин слоты, видит занятость.
3. У занятых слотов видит, кем занято (имя, ссылка на встречу, описание).
4. Берёт публичную ссылку на свой профиль для выкладки.
5. Отменяет любую бронь.

**Посетитель** (аноним по публичной ссылке):
1. Видит месячный календарь с подсветкой дней, где есть свободные слоты.
2. Открывает день, видит свободные и занятые слоты (без указания, кем заняты).
3. Бронирует свободный слот.
4. При брони вводит: имя, ссылку на встречу (обязательно), описание (опционально).

## 3. Доменная модель и правила

- **Доступность**: рабочие часы из конфига (`WORKING_HOURS`, напр. 09:00–18:00 в
  `OWNER_TZ`). Всё свободно по умолчанию; слот занят ⇔ существует бронь.
- **Слоты — сущности в БД**. Генерируются на старте на горизонт «сегодня + месяц» и
  ежедневным идемпотентным шедулером, который досоздаёт слоты на новый день горизонта.
- **Статус слота вычисляется** (не хранится): `PAST` если `start_at < now()`, иначе
  `BOOKED` если есть бронь, иначе `FREE`.
- **Бронь — отдельная сущность** со своим `id` (UUID). Связь слот 1—0..1 бронь.
- **Прошлое** не бронируется, в UI неактивно и не подсвечивается.
- **Горизонт** = сегодня + 1 месяц (`BOOKING_HORIZON_DAYS`). За горизонтом слотов нет.
- **Целостность брони**: проверка «слот уже занят» + `UNIQUE(start_at)` в БД (backstop от
  гонки и дабл-сабмита). Красивая обработка гонки отложена.
- **Отмена**: только владельцем по токену. Посетитель отменять/менять не может.

## 4. Время и часовые пояса

- На проводе (JSON) — `OffsetDateTime` (ISO-8601 со смещением).
- В БД — `timestamptz` (Postgres хранит в UTC).
- Рабочие часы и бизнес-логика — в `OWNER_TZ` (дефолт `Asia/Yerevan`, UTC+4).
  Конвертация границ рабочего дня в UTC — через именованную `ZoneId` (DST-корректно).
- Группировка слотов по дням («какой день у слота», границы запрошенного месяца) —
  **в `OWNER_TZ`**, не в UTC (иначе календарь съезжает на краях).
- Имя зоны (IANA, напр. `Asia/Yerevan`) отдаётся отдельным полем — `OffsetDateTime`
  несёт только смещение, не имя.
- Фронт показывает слоты в TZ браузера и явно отображает, в каком поясе живёт владелец.

## 5. Архитектура

Монорепа, два независимых билда, склейка через Docker:

```
booking/
├─ contract/openapi.yaml      # источник правды контракта (contract-first, рукописный)
├─ backend/                   # Spring Boot + jOOQ (Gradle Kotlin DSL)
├─ frontend/                  # React + Vite + TypeScript (npm)
├─ docs/plans/                # эти планы
├─ compose.yaml              # postgres + app
└─ Dockerfile               # multi-stage: node build → gradle bootJar → JRE runtime
```

Без Nx/Turborepo. Контракт читают обе стороны и не меняют его в рамках реализации.

## 6. Поверхность API (детали — в контракте)

Всё под `/api/v1`. Публичное — без security; админское `/api/v1/admin/**` — Bearer-токен.

Публичные:
- `GET /api/v1/config` → `{ ownerTimeZone, workingHours, slotMinutes, horizonDays }`
- `GET /api/v1/calendar?month=YYYY-MM` → дни месяца со слотами и статусами (без деталей броней)
- `GET /api/v1/slots/{startAt}` → статус слота
- `POST /api/v1/bookings` → создать бронь `{ startAt, name, meetingLink, description? }`

Админские:
- `GET /api/v1/admin/calendar?month=YYYY-MM` → то же + признак/детали занятости
- `GET /api/v1/admin/slots/{startAt}` → полные детали брони
- `DELETE /api/v1/admin/bookings/{id}` → отмена брони

Идентификаторы: слот ↔ `startAt` (UTC-инстант), бронь ↔ `id` (UUID).

## 7. Модель ошибок

RFC 9457 ProblemDetail (`application/problem+json`) + машиночитаемый `errorCode` (enum) +
`errors[]` для валидации. Коды: `422` семантика (прошлое, вне горизонта, кривой слот),
`400` битый запрос/JSON, `409` слот занят, `401` нет/неверный токен, `404` не найдено.

## 8. Конфигурация (env)

- `DB_URL`, `DB_USER`, `DB_PASSWORD`
- `OWNER_TZ` (дефолт `Asia/Yerevan`)
- `WORKING_HOURS` (дефолт `09:00-18:00`)
- `SLOT_MINUTES` (дефолт `30`)
- `BOOKING_HORIZON_DAYS` (дефолт `31`)

Токен владельца в env **не** задаётся: генерируется при первом старте, хранится в БД,
переиспользуется при рестарте, дублируется в лог при каждом старте.

## 9. Тестирование

- Бэк: только интеграционный слой — `@SpringBootTest` + Testcontainers(Postgres) +
  RestAssured/TestRestTemplate; реальный путь HTTP → контроллер → jOOQ → Postgres,
  Flyway-миграции накатываются.
- Фронт: Vitest + React Testing Library + MSW (моки по контракту).
- Валидация полей тестируется на обеих сторонах.
- Playwright (E2E) и CI — отложены.

## 10. Деплой

`compose.yaml`: сервис `postgres` (с volume) + сервис `app`. Spring отдаёт собранный
React как статику (один app-образ, без CORS). Postgres с `healthcheck`, app зависит от
него через `depends_on: condition: service_healthy`. Flyway мигрирует на старте.
У app — `/actuator/health` для healthcheck.

## 11. Отложено / известные ограничения

- Обработка гонки бронирования (есть только UNIQUE-backstop).
- Очистка прошлых слотов из БД.
- Реконсиляция при смене `WORKING_HOURS` после первого старта (конфиг задаётся один раз).
- Rate-limiting публичного `POST /bookings` (риск спама без auth).
- CI/CD, Playwright, линтер контракта, примеры в OpenAPI.
- Email/уведомления, мультивладелец, управление расписанием владельцем.

4. Реализовать фронт по контракту (MSW) до DoD.
