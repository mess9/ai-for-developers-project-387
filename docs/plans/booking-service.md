# План: «Запись на звонок» (Booking Service)

Учебный проект — упрощённая вариация Cal.com для обучения разработке с AI-агентами.
Этот документ — единый источник правды по замыслу. Детали реализации — в
[backend.md](backend.md) и [frontend.md](frontend.md). Контракт API — в `contract/main.tsp`
(TypeSpec; OpenAPI генерится в `contract/tsp-output/openapi.yaml`).

## 1. Суть

Владелец заводит **типы событий** (виды звонков — например «Intro call», 30 мин;
«Demo», 60 мин). Анонимный гость по публичной ссылке выбирает тип, видит свободные
слоты в ближайшие 14 дней и бронирует один из них. Авторизации, личных кабинетов и
интеграций с внешними календарями нет — функциональность намеренно ограничена.

## 2. Акторы и сценарии

**Владелец** (один на инсталляцию, доступ по секретному токену):
1. Создаёт типы событий: название, описание, длительность в минутах.
2. Смотрит страницу предстоящих встреч — единый плоский список броней всех типов.
3. Отменяет любую бронь.

**Гость** (аноним по публичной ссылке):
1. Видит список типов событий (название, описание, длительность).
2. Выбирает тип, открывает его доступность — свободные слоты на ближайшие 14 дней,
   сгруппированные по дням.
3. Бронирует свободный слот.
4. При брони вводит: имя, ссылку на встречу (обязательно), описание (опционально).

## 3. Доменная модель и правила

- **Тип события** — сущность со своим `id` (UUID, генерит сервер): название, описание
  (опц.), длительность в минутах. Длительность кратна 15, в диапазоне 15…120. Сейчас —
  только создание и просмотр (редактирование/удаление отложены, см. §11).
- **Доступность не хранится**, а вычисляется на лету. Рабочие часы из конфига
  (`WORKING_HOURS`, только целые часы, напр. 09:00–18:00 в `OWNER_TZ`) режутся на сетку
  с шагом `GRID_MINUTES` (15). Для типа длительностью `D` слот стартует на каждой
  15-мин границе и существует, если целиком влезает в рабочие часы (`start+D ≤ end`).
- **Слот идентифицируется парой** (`eventTypeId`, `startAt`): без типа неизвестна
  длительность, а значит и интервал `[startAt, startAt+D)`.
- **Свободен** ⇔ интервал слота не пересекается ни с одной существующей бронью.
  Прошлые старты и старты вне окна 14 дней в выдачу не попадают.
- **Бронь — отдельная сущность** со своим `id` (UUID): ссылка на тип, `start_at`,
  `end_at` (хранится у брони — прошлые встречи не «едут» при изменении типа), данные
  гостя. Денормализованное название типа отдаётся в выдаче для читаемости.
- **Правило занятости**: на одно и то же время нельзя две брони — **даже разных типов**.
  Формально — интервалы броней не пересекаются. Проверяется на бэке overlap-запросом
  перед вставкой (чистый 409) и подстраховано БД-констрейнтом `EXCLUDE USING gist` от гонки
  (нарушение → 409). Полноценный UX дабл-сабмита отложен (§11).
- **Горизонт** = сегодня + 14 дней (`BOOKING_HORIZON_DAYS`). За горизонтом слотов нет.
- **Отмена**: только владельцем по токену. Гость отменять/менять не может.

## 4. Время и часовые пояса

- На проводе (JSON) — `OffsetDateTime` (ISO-8601 со смещением).
- В БД — `timestamptz` (Postgres хранит в UTC).
- Рабочие часы и бизнес-логика — в `OWNER_TZ` (дефолт `Asia/Yerevan`, UTC+4).
  Конвертация границ рабочего дня в UTC — через именованную `ZoneId` (DST-корректно).
- Группировка слотов по дням («какой день у слота», границы окна 14 дней) —
  **в `OWNER_TZ`**, не в UTC (иначе дни съезжают на краях).
- Имя зоны (IANA, напр. `Asia/Yerevan`) отдаётся отдельным полем — `OffsetDateTime`
  несёт только смещение, не имя.
- Фронт показывает слоты в TZ браузера и явно отображает, в каком поясе живёт владелец.

## 5. Архитектура

Монорепа, два независимых билда, склейка через Docker:

```
booking/
├─ contract/main.tsp          # источник правды контракта (TypeSpec, design-first)
│                             #   → tsp-output/openapi.yaml (генерится, не коммитим)
├─ backend/                   # Spring Boot + jOOQ (Gradle Kotlin DSL)
├─ frontend/                  # React + Vite + TypeScript (npm)
├─ docs/plans/                # эти планы
├─ compose.yaml              # postgres + app
└─ Dockerfile               # multi-stage: node build → gradle bootJar → JRE runtime
```

Без Nx/Turborepo. Контракт читают обе стороны и не меняют его в рамках реализации.

## 6. Поверхность API (детали — в контракте)

Всё под `/api/v1`. Публичное — без security; админское `/api/v1/admin/**` — Bearer-токен.

Публичные:
- `GET /api/v1/config` → `{ ownerTimeZone, workingHours, gridMinutes, horizonDays }`
- `GET /api/v1/event-types` → список типов `{ id, name, description?, durationMinutes }`
- `GET /api/v1/event-types/{id}/availability` → окно 14 дней от сегодня: **все дни окна**
  (день без свободных слотов — с флагом `hasFreeSlots:false` и пустым `slots`), в `slots`
  только bookable старты выбранного типа (`startAt`+`endAt`)
- `POST /api/v1/bookings` → создать бронь
  `{ eventTypeId, startAt, name, meetingLink, description? }`

Админские:
- `POST /api/v1/admin/event-types` → создать тип `{ name, description?, durationMinutes }`
- `GET /api/v1/admin/event-types` → список типов (для владельца)
- `GET /api/v1/admin/bookings` → предстоящие встречи: плоский список броней всех типов
- `DELETE /api/v1/admin/bookings/{id}` → отмена брони

Идентификаторы: тип события ↔ `id` (UUID); слот ↔ пара (`eventTypeId`, `startAt`);
бронь ↔ `id` (UUID).

## 7. Модель ошибок

RFC 9457 ProblemDetail (`application/problem+json`) + машиночитаемый `errorCode` (enum) +
`errors[]` для валидации. Коды: `422` семантика (прошлое, вне горизонта, старт не из
сетки), `400` битый запрос/JSON/валидация полей, `409` интервал пересекается с бронью,
`401` нет/неверный токен, `404` не найдено (тип события или бронь).

## 8. Конфигурация (env)

- `DB_URL`, `DB_USER`, `DB_PASSWORD`
- `OWNER_TZ` (дефолт `Asia/Yerevan`)
- `WORKING_HOURS` (дефолт `09:00-18:00`, только целые часы)
- `GRID_MINUTES` (дефолт `15`) — шаг сетки стартов
- `BOOKING_HORIZON_DAYS` (дефолт `14`)

Токен владельца в env **не** задаётся: генерируется при первом старте, хранится в БД,
переиспользуется при рестарте, дублируется в лог при каждом старте.

## 9. Тестирование

- Бэк: только интеграционный слой — `@SpringBootTest` + Testcontainers(Postgres) +
  RestAssured/TestRestTemplate; реальный путь HTTP → контроллер → jOOQ → Postgres,
  Flyway-миграции накатываются.
- Фронт: Vitest + React Testing Library + MSW (моки по контракту).
- Валидация полей тестируется на обеих сторонах.
- Playwright (E2E) и CI — отложены.

## 10. Деплой

`compose.yaml`: сервис `postgres` (с volume) + сервис `app`. Spring отдаёт собранный
React как статику (один app-образ, без CORS). Postgres с `healthcheck`, app зависит от
него через `depends_on: condition: service_healthy`. Flyway мигрирует на старте.
У app — `/actuator/health` для healthcheck.

## 11. Отложено / известные ограничения

- UX-обработка дабл-сабмита (БД-защита от гонки уже есть — `EXCLUDE`-констрейнт; речь о
  дружелюбном поведении клиента/идемпотентности на повторную отправку).
- Редактирование и удаление типов событий (что делать с уже созданными бронями типа).
- Реконсиляция при смене `WORKING_HOURS` после первого старта (конфиг задаётся один раз).
- Rate-limiting публичного `POST /bookings` (риск спама без auth).
- CI/CD, Playwright, линтер контракта, примеры в OpenAPI.
- Email/уведомления, мультивладелец, управление расписанием владельцем.

## 12. Порядок работ

1. Зафиксировать план (этот документ) — сделано.
2. Написать контракт в `contract/main.tsp` (TypeSpec → OpenAPI).
3. Реализовать бэк (эталон поведения) до DoD.
4. Реализовать фронт по контракту (MSW) до DoD.
5. Собрать `compose`, проверить сквозной сценарий вручную.
6. (Позже) CI, E2E, раскатка.

Definition of Done по каждой сессии — в `CLAUDE.md`.
