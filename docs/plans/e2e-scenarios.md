# E2E-сценарии (Playwright)

Сквозные сценарии для будущих Playwright-тестов, интегрируемых в CI/CD. Покрывают
ключевые пути гостя и владельца на реальном стеке (фронт + бэк + Postgres, поднятые
через `compose.yaml`). Домен и правила — в [booking-service.md](booking-service.md),
экраны — в [frontend.md](frontend.md).

## Стратегия тестов

- **Язык/модуль:** тесты на TypeScript, в отдельном модуле репозитория (напр. `e2e/`),
  независимом от `frontend/` и `backend/`. Свой `package.json`, Playwright и
  pg-клиент (напр. `pg`) для прямого доступа к БД.
- **Предусловия — прямой посев в БД.** Нужные данные (типы событий, существующие брони)
  тест кладёт **напрямую в Postgres** SQL-вставками, а не через API. Это даёт контроль
  над `id`, временем и состоянием без зависимости от админских endpoint'ов.
- **Действия — через интерфейс (приоритет).** Основной прогон сценария — клики и ввод
  в UI как у реального пользователя. API дёргаем напрямую только там, где это явно нужно
  (эмуляция гонки, недоступный через UI кейс).
- **Проверки — UI в приоритете, БД как подтверждение.** Сначала проверяем то, что видит
  пользователь (тосты, списки, исчезновение слота). Состояние, не отражённое в UI
  однозначно (факт записи в БД, поля брони, факт удаления), **дочитываем из Postgres**.
- **Доступ к БД:** тест-модуль ходит в ту же базу, что и `app` (параметры подключения —
  из env, согласованы с `compose.yaml`). Хелперы: `seed*` (вставка), `readBookings`/
  `readEventTypes` (чтение), `truncateAll` (сброс).

## Базовые предусловия (для всех сценариев)

- Поднято приложение: `app` + `postgres` (Flyway-миграции применены), БД доступна
  тест-модулю напрямую.
- БД сбрасывается перед каждым сценарием (`truncateAll`) — изоляция тестов; данные не
  «текут» между сценариями.
- Токен владельца **читаем напрямую из БД** (универсально, не зависит от формата лога) —
  тест берёт его и кладёт в `localStorage` перед заходом в админку.
- Часовой пояс владельца — дефолтный `Asia/Yerevan`, рабочие часы `09:00–18:00`,
  горизонт 14 дней. Тесты не зависят от пояса CI-раннера: время сверяется относительно
  `ownerTimeZone` из `GET /config`, а не от локали машины.

> **Замечание про время.** Доступность вычисляется от «сегодня», слоты — только будущие.
> Чтобы тесты были детерминированы, выбираем слот не «первый попавшийся», а вычисляемый:
> например, ближайший свободный слот первого дня окна, у которого `hasFreeSlots:true`.

## Сценарий 1 — Гость бронирует слот (happy path)

**Цель:** базовый путь записи от списка типов до подтверждения.

**Предусловие (посев в БД):** вставить один тип события (напр. «Intro call», 30 мин)
прямо в таблицу типов.

**Шаги:**
1. Гость открывает публичную страницу `/`.
2. Видит баннер владельца (часовой пояс + рабочие часы) и список типов событий.
3. Выбирает тип → видит доступность: 14-дневное окно, дни с `hasFreeSlots:true`
   кликабельны, дни без слотов погашены.
4. Кликает по доступному дню → видит список свободных слотов; у каждого время показано
   в двух поясах (пояс владельца + «у вас это HH:mm»).
5. Выбирает слот → заполняет форму: имя, ссылка на встречу (валидный URL), описание (опц.).
6. Отправляет форму.

**Ожидаемый результат:**
- *(UI)* Тост «Вы записаны».
- *(UI)* Возврат к экрану доступности и рефетч: только что забронированный слот **исчез**
  из списка свободных.
- *(БД)* В таблице броней появилась ровно одна запись с введёнными именем, ссылкой,
  `event_type_id` и `start_at`/`end_at` выбранного слота.

## Сценарий 2 — Владелец создаёт тип события, гость его видит

**Цель:** сквозная связка «владелец завёл тип → он доступен гостю».

**Предусловие:** типов нет (чистая БД после `truncateAll`). Здесь тип создаётся
**через UI** намеренно — это и есть проверяемый путь владельца.

**Шаги:**
1. Владелец заходит в `/admin` (токен уже в `localStorage`).
2. В секции «Типы событий» заполняет форму: название, описание, длительность
   (`select` из кратных 15, напр. 30 мин) → отправляет.
3. Новый тип появляется в списке типов владельца.
4. Гость открывает `/` (новая вкладка/контекст).

**Ожидаемый результат:**
- *(UI)* Новый тип виден в списке владельца и в публичном списке у гостя — с правильными
  названием, описанием и длительностью.
- *(UI)* При выборе типа доступность считается под его длительность (слоты влезают в
  рабочие часы: `start + D ≤ end`).
- *(БД)* В таблице типов появилась одна запись с заданными полями.

## Сценарий 3 — Владелец видит бронь и отменяет её, слот снова свободен

**Цель:** связка «гость забронировал → владелец видит и отменяет → слот возвращается».

**Предусловие (посев в БД):** вставить тип события и активную бронь на конкретный слот
будущего дня (имя, ссылка, `start_at`/`end_at`).

**Шаги:**
1. Владелец заходит в `/admin`, секция «Предстоящие встречи».
2. Видит бронь в плоском списке: имя гостя, ссылка, описание, название типа, время
   (в двух поясах).
3. Нажимает «Отменить» у этой брони.
4. (Проверка возврата слота) Гость открывает доступность того же типа на тот же день.

**Ожидаемый результат:**
- *(UI)* После отмены бронь пропадает из списка предстоящих встреч.
- *(UI)* Ранее занятый слот снова присутствует среди свободных в доступности.
- *(БД)* Запись брони удалена из таблицы.

## Сценарий 4 — Занятое время недоступно (в т.ч. для другого типа)

**Цель:** правило занятости — на одно время не больше одной брони, даже разных типов.

**Предусловие (посев в БД):** вставить два типа разной длительности (напр. 30 и 60 мин)
и одну бронь на конкретный слот.

**Шаги:**
1. Гость открывает доступность **другого** типа на тот же день.
2. Проверяет, что слоты, чьи интервалы пересекаются с уже существующей бронью,
   в выдаче отсутствуют.
3. (Гонка / устаревшая выдача) Гость держит открытой форму на свободном слоте; тест
   **вставляет в БД бронь на это же время** (эмуляция «заняли после загрузки»), затем
   гость сабмитит форму.

**Ожидаемый результат:**
- *(UI)* Пересекающиеся слоты не показываются — выдача содержит только реально свободные
  старты.
- *(UI)* При сабмите на уже занятое время — тост «время только что заняли»
  (`409 SLOT_ALREADY_BOOKED`), доступность перезапрашивается.
- *(БД)* Второй (конфликтующей) брони не появилось — запись осталась одна.

## Сценарий 5 — Валидация формы брони и защита админки

**Цель:** ошибки ввода гостя и контроль доступа владельца.

**Предусловие (посев в БД):** вставить один тип события (для прохода к форме брони).

**Часть A — валидация формы гостя:**
1. Гость доходит до формы брони (тип → день → слот).
2. Пытается отправить с пустым именем / без ссылки на встречу / с невалидным URL.

**Ожидаемый результат A:**
- Submit заблокирован клиентской валидацией, ошибки показаны у полей; запрос на бэк не
  уходит (или, при обходе клиента, `400 VALIDATION_FAILED` мапится на поля).

**Часть B — доступ в админку:**
1. Заход в `/admin` без токена (пустой `localStorage`) → показана форма входа, защищённые
   данные не подгружаются.
2. Заход с неверным токеном → API отвечает `401 UNAUTHORIZED`.

**Ожидаемый результат B:**
- При `401` токен сбрасывается из `localStorage`, происходит возврат к форме входа.
- С валидным токеном — секции «Типы событий» и «Предстоящие встречи» загружаются.
